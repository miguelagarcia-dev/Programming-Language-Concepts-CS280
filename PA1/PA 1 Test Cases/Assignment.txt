Programming Assignment 1
Lexical Analyzer for the Basic Pascal-Like Language
Total Points: 20
In this programming assignment, you will be building a lexical analyzer for a programming
language, referred to as Basic Pascal-Like, and a program to test it. This assignment will be
followed by two other assignments to build a parser and an interpreter for the Basic Pascal-Like
language. Although, we are not concerned about the syntax definitions of the language rules in this
assignment, we intend to introduce them ahead of Programming Assignment 2 in order to
determine the definitions of the language lexical units (i.e., terminals): reserved words, constants,
identifier(s), and operators. The Basic Pascal-Like language includes some basic syntactic and
sematic characteristics from the well-known Pascal programming language, that explains the use
of a name for this language. The syntax definitions of the language are given below using EBNF
notations. However, the details of the meanings (i.e. semantics) of the language constructs will be
given later on.
1. Prog ::= PROGRAM IDENT ; Block .
2. Block ::= [ DeclPart ] CompStmt
3. DeclPart ::= [ ConstPart ] [ VarPart ]
4. ConstPart ::= CONST ConstDef { ; ConstDef } ;
5. ConstDef ::= IDENT = Expr
6. VarPart ::= VAR DeclStmt { ; DeclStmt } ;
7. DeclStmt ::= IDENT {, IDENT } : Type [:= Expr]
8. Type ::= INTEGER | REAL | BOOLEAN | CHAR | STRING
9. Stmt ::= SimpleStmt | StructuredStmt
10. SimpleStmt ::= AssignStme | ReadLnStmt | WriteLnStmt | WriteStmt
11. StructuredStmt ::= IfStmt | CompStmt
12. CompStmt ::= BEGIN Stmt {; Stmt } END
13. WriteLnStmt ::= WRITELN ( ExprList )
14. WriteStmt ::= WRITE ( ExprList )
15. IfStmt ::= IF Expr THEN Stmt [ ELSE Stmt ]
16. AssignStmt ::= Variable := Expr
17. Variable ::= IDENT
18. ReadLnStmt ::= ReadLnStmt ( VarList )
19. ExprList ::= Expr { , Expr }
20. VarList ::= Variable {, Variable }
21. Expr ::= RelExpr ::= SimpleExpr [ ( = | < | > ) SimpleExpr ]
22. SimpleExpr :: Term { ( + | - | OR ) Term }
23. Term ::= SFactor { ( * | / | DIV | MOD | AND ) SFactor }
24. SFactor ::= [( - | + | NOT )] Factor
25. Factor ::= IDENT | ICONST | RCONST | SCONST | BCONST | CCONST | (Expr)

Based on the language definitions, the lexical rules of the language and the assigned tokens to the
terminals are as follows:
1. The language has identifiers, referred to by IDENT terminal, which are defined as a word that
starts by a letter, and followed by zero or more letters, digits, underscores ‘_’, or ‘$’ characters.
Note that all identifiers are case sensitive. It is defined as:
IDENT ::= Letter ( Letter | Digit | _ | $ )*
Letter ::= [a-z A-Z]
Digit ::= [0-9]
2. Integer constant is referred to by ICONST terminal, which is defined as one or more digits. It
is defined as:
ICONST ::= Digit+
3. Real constant is a fixed-point real number referred to by RCONST terminal, which is defined
as one or more digits followed by a decimal point (dot) and one or more digits, followed by an
optional exponent. It is defined as:
RCONST ::= Digit+ Fraction Exponent?
Fraction ::= \. Digit+
Exponent ::= ( E | e ) ( + | - )? Digit+
For example, real number constants such as 12.0, 0.2, 2.0, 3.0E-2 and 3.0e2 are accepted as
real constants, but .2, 3., and 2.45.2 are not. Note that “.2” is recognized as a DOT followed
by the integer constant 2.
4. A string literal is referred to by SCONST terminal, which is defined as a sequence of characters
delimited by single quotes, that should all appear on the same line. For example,
‘Hello to CS 280.’ is a string literal. While, “Hello to CS 280.” Or ‘Hello
to CS 280.” are not.
5. Boolean constant is referred to by the BCONST terminal, which is defined by either of the
constant values “true” or “false”. Note that the “true” and “false are treated as keywords.
BCONST ::= (TRUE | FALSE )

9. A comment is defined by one of the following format:
• {This is a comment. }
• (*This is a comment. *)
Comments may overlap more than one line, as multi-line comments. The symbols used to
delimit a comment must be of the kind that match.
10. White spaces are skipped. However, white spaces between tokens are used to improve
readability and can be used as a one way to delimit tokens.
11. An error will be denoted by the ERR token.
12. End of file will be denoted by the DONE token.
Lexical Analyzer Requirements:
A header file, lex.h, is provided for you. It contains the definitions of the LexItem class, and
an enumerated type of token symbols, called Token, and the definitions of three functions to be
implemented. These are:
extern ostream& operator<<(ostream& out, const LexItem& tok);
extern LexItem id_or_kw(const string& lexeme, int linenum);
extern LexItem getNextToken(istream& in, int& linenum);
You MUST use the header file that is provided. You may NOT change it.
I. You will write the lexical analyzer function, called getNextToken, in the file “lex.cpp”.
The getNextToken function must have the following signature:
LexItem getNextToken (istream& in, int& linenumber);
The first argument to getNextToken is a reference to an istream object that the function
should read from. The second argument to getNextToken is a reference to an integer that
contains the current line number. getNextToken should update this integer every time it
reads a newline from the input stream. getNextToken returns a LexItem object. A
LexItem is a class that contains a token, a string for the lexeme, and the line number as data
members.
Note that the getNextToken function performs the following:
1. Any error detected by the lexical analyzer should result in a LexItem object to be returned
with the ERR token, along with the object’s lexeme data member including an error
message appended with the string in which the error was detected. The following table
describes the Error Types to be detected by the Lexical Analyzer, and the corresponding
error messages that should be returned for those types by an ERR token LexItem object.
Error Type Message
Invalid character for starting a token "Invalid character for starting a token \"" + lexeme + "\""
New line within a string literal "New line is not allowed within string literal \"" + lexeme + "\""
Incorrect exponent for a floating-point constant "Invalid exponent for a floating-point constant \"" + lexeme+ "\""
Invalid floating-point constant "Invalid floating-point constant \"" + lexeme + "\""
Comment missing closing symbol(s) error "Missing closing symbol(s) for a comment \"" + lexeme+ "\""
Comment nesting error "Invalid nesting of comments \"" + lexeme + "\""
Where, “lexeme” refers to the string part that caused the error in the input file.
2. Note also that both ERR and DONE are unrecoverable. Once the getNextToken
function returns a LexItem object for either of these tokens, you shouldn’t call
getNextToken again. 
3. Tokens may be separated by spaces, but in most cases are not required to be. For example,
the input characters “3+7” and the input characters “3 + 7” will both result in the
sequence of tokens ICONST PLUS ICONST. Similarly, The input characters
‘Hello’ ‘World’, and the input characters ‘Hello’’World’
will both result in the token sequence SCONST SCONST.
II. You will implement the id_or_kw() function. The function accepts a reference to a string of a
general identifier lexeme (i.e., keyword, IDENT) and a line number and returns a LexItem
object. It searches for the lexeme in a directory that maps a string value of a keyword to its
corresponding Token value, and it returns a LexItem object containing the keyword Token if
it is found. Otherwise, it returns a LexItem object containing an IDENT token.
III. You will implement the overloaded operator function operator<<(). The operator<<
function accepts a reference to an ostream object and a reference to a LexItem object, and
returns a reference to the ostream object. The operator<< function should print out the
string value of a Token contained in the referenced tok parameter of the function, followed by
its lexeme, using the format shown in the following table.

comments “{“

Note, if the Token is an ERR, it will print out the following:
ERR: Error in line (N) error-message
where N is the line number in which the error is found in the input file. The error-message
refers to the contents of an error message included in the lexeme data member of the LexItem
object passed back by the lexical analyzer. Note: All printed out messages should be followed
by a newline.
Testing Program Requirements:
It is recommended to implement the lexical analyzer in one source file, and the main test
program in another source file. The testing program is a main() function that takes several
command line flags. The notations for input flags are as follows: 
● -all (optional): if present, every token is printed out when it is seen according to the format
described for printing out each type of a token in point III above.
● -num (optional): if present, prints out all the unique integer and real constants in two
separate lists, where values of each category are displayed in numeric order.
● -str (optional): if present, prints out all the unique string constants in order.
● -ids (optional): if present, prints out all of the unique identifiers and keywords in two
separate lists, where names of each category are displayed in order.
● filename argument must be passed to main function. Your program should open the file
and read from that filename.
Note, your testing program should apply the following rules:
1. The flag arguments (arguments that begin with a dash) may appear in any order, and may
appear multiple times. Only the last appearance of the same flag is considered.
2. There can be at most one file name specified on the command line. If more than one filename
is provided, the program should print on a new line the message “ONLY ONE FILE NAME
IS ALLOWED.” and it should stop running. If no file name is provided, the program should
print on a new line the message “NO SPECIFIED INPUT FILE.”. Then the program should
stop running.
3. If an unrecognized flag is present, the program should print on a new line the message
“UNRECOGNIZED FLAG {arg}”, where {arg} is whatever flag was given. Then the
program should stop running.
4. If the program cannot open a filename that is given, the program should print on a new line
the message “CANNOT OPEN THE FILE arg”, where arg is the filename given. Then the
program should stop running.
5. If getNextToken function returns ERR, the program should display error message
according to the description given for printing the ERR token in point III above. For
example, a file that contains an invalid real constant, as 27.57.5, in line 2 of the file, the
program should print the message:
ERR: Error in line (2) Invalid floating-point constant "27.57.5 "
6. The program should repeatedly call getNextToken until it returns DONE or ERR. If it
returns DONE, the program prints the summary information, then handles the flags based on
the order: “-ids”, “-num”, and “-str”. The summary information are as follows:
Lines: L
Total Tokens: M
Identifiers & Keywords: N
Numbers: O
Booleans: P
Strings: Q
Where L is the number of input lines, M is the total number of tokens (not counting DONE),
N is the total number of identifiers and keywords tokens, O is the total number of integer and
real constants tokens, P is the number of Boolean constants token, and Q is the number of
string literals token. If the file is empty the following output message is displayed:
Empty File.
7. If the -all option is present, the program should print out each token as it is read and
recognized, one token per line. The formats of printed tokens are as explained above in point
III for the overloaded operator function, operator<<.
8. The -str option should cause the program to print out the label “STRINGS:” on a line by
itself, followed by every unique string constant delimited by single quotes, followed by the
count of its occurrence delimited by parentheses, in alphabetical order and separated by
commas. String literals are separated by commas. If there are no SCONSTs in the input, then
nothing is printed.
9. The -num option should cause the program to print out the lists of all integer and real
constants. Each list should start by a header as “INTEGERS:” or “REALS:” on a line by
itself, followed by the list of the numeric constants and the count of their occurrences
between parentheses, in order and separated by commas. If there are no NCONSTs in the
input, then nothing is printed.
10. The -ids option should cause the program to print out the lists of all identifiers and keywords.
Each list should start by a header as “IDENTIFIERS:” or “KEYWORDS:” on a line by itself,
followed by the list of the named identifiers or keywords and the count of their occurrences
between parentheses, in order and separated by commas. If there are no identifiers in the
input, then nothing is printed.
